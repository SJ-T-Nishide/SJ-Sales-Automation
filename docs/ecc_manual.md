## はじめに：ECCを入れると何が変わるか

Claude Codeをそのまま使うのは「優秀だが何も知らない新入社員」を雇うようなものです。  
ECCを入れると「60人の専門家チームを持つ会社」になります。

| Before（ECC導入前） | After（ECC導入後） |
|--------------------|--------------------|
| 何でも1人でやるClaude | 専門家に自動で仕事を振り分ける |
| 毎回ゼロから説明が必要 | 前回の会話を覚えている |
| コストが読めない | トークン使用量を自動で最適化 |
| 使うたびに同じミスをする | 使えば使うほど賢くなる |

## 第1部：エージェント（60人の専門家チーム）

### エージェントとは何か

通常のClaudeは「何でも屋」です。エージェントは「特定分野の専門家」です。  
Claude Codeが「この仕事はこの専門家に任せよう」と**自動で判断して呼び出します**。  
あなたが明示的に呼ぶ必要はありません。

### 主要エージェント一覧（御社で特に使えるもの）

#### 計画・設計系

| エージェント | 日本語 | いつ動くか | 何をしてくれるか |
|------------|--------|-----------|----------------|
| `planner` | 計画担当 | 大きな仕事を頼んだとき | 実装前に詳細な手順書を作る |
| `architect` | 設計担当 | システムを作るとき | 全体設計・依存関係・リスクを整理する |
| `chief-of-staff` | 参謀 | 複数タスクが絡むとき | 優先順位をつけてタスクを整理する |

#### コード品質系

| エージェント | 日本語 | いつ動くか | 何をしてくれるか |
|------------|--------|-----------|----------------|
| `python-reviewer` | Pythonレビュー担当 | Pythonファイルを変更したとき | コードの問題点を指摘する |
| `security-reviewer` | セキュリティ担当 | APIキーやログイン処理を触るとき | セキュリティの穴を検出する |
| `code-reviewer` | コードレビュー担当 | コードを書いたとき | 品質・保守性を評価する |
| `tdd-guide` | テスト担当 | 重要な処理を書くとき | テストの書き方を指導する |
| `silent-failure-hunter` | バグ探偵 | 「なんか動いてるけど変」なとき | 静かに失敗しているバグを探す |

#### 調査・最適化系

| エージェント | 日本語 | いつ動くか | 何をしてくれるか |
|------------|--------|-----------|----------------|
| `performance-optimizer` | 速度改善担当 | 処理が遅いとき | ボトルネックを特定して改善策を提案 |
| `seo-specialist` | SEO担当 | LP・ページを作るとき | 検索流入を増やす観点でレビュー |
| `doc-updater` | ドキュメント担当 | コードを変えたとき | README・マニュアルを自動更新 |

#### ビルドエラー解決系

| エージェント | 日本語 | いつ動くか | 何をしてくれるか |
|------------|--------|-----------|----------------|
| `build-error-resolver` | エラー修理担当 | エラーが出たとき | エラーの原因を特定して最小限の修正をする |

### エージェントを手動で呼ぶ方法

自動で呼ばれることが多いですが、明示的に指定もできます。

```
例：「planner エージェントとして、このシステムの実装計画を立ててください」
例：「security-reviewer として、このコードにAPIキーの漏れがないか確認して」
```

## 第2部：コマンド（ワンタッチで呼び出す75個の指示）

`/コマンド名` と入力するだけで実行できます。

### よく使うコマンド TOP10

| コマンド | 使い方 | 何が起きるか |
|---------|--------|------------|
| `/plan` | `/plan 営業自動化システムを作りたい` | 実装手順書を自動生成 |
| `/code-review` | コードを書いた後に実行 | セキュリティ・品質をチェック |
| `/security-scan` | `/security-scan` | APIキー漏れ・脆弱性を全ファイルスキャン |
| `/build-fix` | エラーが出たら実行 | エラーを自動で解析・修正 |
| `/update-docs` | コードを変えたら実行 | ドキュメントを最新化 |
| `/cost-report` | 月末に実行 | Claude API使用コストを確認 |
| `/model-route` | 重い処理の前に実行 | タスクに最適なモデルを提案 |
| `/refactor-clean` | 定期的に実行 | 不要なコードを検出・整理 |
| `/resume-session` | 作業を再開するとき | 前回の作業内容を呼び出す |
| `/save-session` | 作業を中断するとき | 現在の状態を保存する |

### 並列実行コマンド（複数の作業を同時にやらせる）

| コマンド | 使い方 |
|---------|--------|
| `/multi-plan` | 複数の作業を同時に計画させる |
| `/multi-execute` | 複数の作業を同時に実行させる |

## 第3部：裏で自動的に動いている機能（Hooks）

何も操作しなくても、Claudeが作業するたびに自動で動いています。

### 6つの自動番人

| いつ動くか | 何をしているか |
|-----------|--------------|
| **会話を始めるとき** | 前回のセッション要約を自動で読み込む。「前回どこまでやったか」を覚えている |
| **ツールを使う前** | 作業内容を観察して「学習データ」として蓄積する |
| **ツールを使う前** | 50回ツールを使ったら「そろそろコンテキストを整理しましょう」と提案する |
| **コンテキスト圧縮前** | 圧縮される前の状態を保存する |
| **会話が終わるとき** | その日の作業内容（変更ファイル・使ったツール・会話要約）を自動保存 |
| **会話が終わるとき** | 10回以上のやりとりがあれば、そのセッションから「学び」を抽出する |

### セッションをまたいだ記憶の仕組み

```
今日のセッション終了
    ↓
~/.claude/session-data/ に自動保存
    ↓
明日のセッション開始
    ↓
「昨日はECC導入を完了しました。次はCommandsの使い方を試してみましょう」
と自動で教えてくれる
```

## 第4部：外部ツール連携（MCP）

現在接続している3つのサービス。

### sequential-thinking（段階的思考）
**何か：** 複雑な問題を解くとき、Claude が自動で「ステップ1、ステップ2...」と思考を整理するようになります。  
**使い方：** 意識しなくてOK。難しい判断をお願いしたとき自動で発動します。  
**御社での活用例：** 「この物件は民泊向きか投資向きか判断して」のような多角的な判断

### token-optimizer（トークン削減）
**何か：** Claudeとのやりとりのコストを最大95%削減できる機能です。  
**使い方：** 意識しなくてOK。重複した内容や不要な情報を自動で圧縮します。  
**御社での活用例：** 長い契約書や物件データを扱うときのコスト削減

### exa-web-search（高精度Web検索）
**何か：** 通常のGoogle検索より精度の高い、AI向け専用検索エンジンです。  
**使い方：** 調査を依頼するときに「exa で検索して」と言うだけ。  
**御社での活用例：**

```
「exaで大阪市の民泊規制の最新情報を調べて」
「exaでAirDNAの2026年大阪の市場データを探して」
「exaで競合する民泊運営会社のサービス内容を調べて」
```

## 第5部：使えば使うほど賢くなる仕組み（継続的学習）

### instinct（直感）システム

使うたびに「このユーザーはこういう判断をする」というパターンをYAMLファイルに蓄積します。

保存先：`~/.local/share/ecc-homunculus/`

**蓄積される「直感」の例：**
- 「このユーザーはGASのasync/awaitを使わない（理由：GASは同期実行）」
- 「このユーザーはAPIキーを.envで管理する」
- 「このユーザーは変更前に必ず確認を取る」

### 確認コマンド

```
/instinct-status   → 現在蓄積されたinstinctの一覧を見る
/evolve            → instinctを手動で更新・進化させる
```

## 第6部：実際の使い方シナリオ（御社業務別）

### シナリオ1：新しい機能を作りたいとき

```
1. 「〇〇機能を作りたい。まず計画を立てて」
   → planner エージェントが自動で動く
   
2. 計画を確認してOKなら「実装して」
   → architectエージェントが設計レビュー
   → 実装が始まる
   
3. 実装後「レビューして」
   → code-reviewer・security-reviewer が自動チェック
   
4. 問題があれば「修正して」
   → build-error-resolver が対応
```

### シナリオ2：競合・市場調査をしたいとき

```
「exaで2026年の大阪民泊市場の現状を調べて、
 投資家向けに以下をまとめて：
 ・稼働率の推移
 ・競合施設数
 ・規制の変化
 ・AirDNAデータで見る単価トレンド」
```

### シナリオ3：コードが動かなくなったとき

```
エラーメッセージをそのまま貼り付けて「直して」と言うだけ
→ build-error-resolver が自動で原因を特定・最小限の修正を提案
```

### シナリオ4：営業資料・LP・セミナー原稿を作りたいとき

```
「民泊経営パッケージのLPの文章を作って。
 ターゲット：不動産投資に興味がある40代会社員
 訴求ポイント：手離れの良さ・安定利回り
 seo-specialist としてSEO観点も入れて」
```

### シナリオ5：作業を中断・再開するとき

```
中断前：「/save-session」
        → 作業内容が自動保存される

翌日：  「/resume-session」
        → 「昨日は〇〇まで完了しました。次は△△です」と表示される
```

## 第7部：覚えておくべき基本ルール

### やってはいけないこと

| NG操作 | 理由 |
|--------|------|
| `reins_tool/config.ini` を変更する | REINS認証情報が入っている |
| `.env`ファイルを変更する | APIキーが入っている |
| `contract_generator/` を無断で変更する | 本番稼働中の契約書生成ツール |

### Claudeへの上手な指示の出し方

**悪い例：**
```
「これ直して」（何を直すかわからない）
```

**良い例：**
```
「この Python スクリプトを実行したら以下のエラーが出ました。
 原因を特定して最小限の修正をしてください。
 [エラーメッセージをここに貼る]」
```

**ポイント：**
1. **何を・どうしたいか** を明確に
2. エラーは**そのまま全文貼る**
3. 大きな作業は「まず計画して、確認してから実装」の順で
4. 本番コード（`contract_generator`等）は「変更前に確認して」と一言添える

## 第8部：コスト管理

### 月々の確認方法

```
/cost-report   → 今月のAPI使用コストを確認
```

### コストを下げるコツ

1. **長い会話は適度に区切る** → `/save-session` で保存して新しいセッションで再開
2. **重要でない作業はHaikuで** → 「Haikuで実行して」と指定（安い）
3. **同じ内容を何度も聞かない** → `MEMORY.md` に保存しておく
