# Success Japan マーケティングオートメーションシステム 全体概要
**対象読者**: マーケティング担当者・営業担当者  
**最終更新**: 2026-06-16  
**作成者**: 西出高宏 / AI設計

---

## このシステムを一言で言うと

> **「リードを取りこぼさず、自動で追い続け、営業が動くべき時だけアラートを出すシステム」**

UTAGEやSalesforceのような市販ツールを使わず、Success Japan専用に自社で構築。  
将来的には外部企業へのSaaS提供も視野に入れた設計になっています。

---

## なぜ自社で作るのか

| 比較軸 | UTAGE（市販ツール） | 自社システム |
|--------|-------------------|------------|
| 導入スピード | すぐ使える | 数週間かかる |
| 月額コスト | ¥17,600 | ¥22,000〜 |
| データの所有 | ベンダー管理（取り出しにくい） | 完全自社管理 |
| AIマーケ提案 | できない | **できる（Phase 4）** |
| SaaS外販 | できない | **できる（Phase 5）** |
| カスタマイズ | 限界あり | 無制限 |

**結論**: 短期コストはほぼ同じ。でも「AIが施策提案する」「外販してビジネス化する」という2つの目標はUTAGEでは絶対に達成できないため、自社構築を選択。

## システム全体の流れ（図解）

```
【集客・流入】
ゴールドオンライン / 楽待 / 健美家 / Meta広告 / 紹介
        ↓
【申込・登録】
selecttype（セミナー申込フォーム）
        ↓
【リードDB】← ここが全ての中心
Supabase（クラウドデータベース・東京）
  ├─ 名前・メール・電話・流入元
  ├─ 熱度（A/B/C）・ステータス・担当者
  └─ 全送信履歴・アクションログ
        ↓
【自動アクション】（GASが5分ごとにチェック）
  ├─ メール自動送信（Resend）
  ├─ SMS自動送信（ValueSMS）
  ├─ LINE自動送信（Phase 2〜）
  └─ Slackアラート → 営業担当へ通知
        ↓
【人間のアクション】（クロージングのみ）
西出さん・松村さんが電話・面談
        ↓
【成約・失注】
ステータス更新 → 次のシーケンスへ
```

## 現在のリードDB（Supabase）の構造

リード1件に記録される情報：

| 項目 | 内容 | 例 |
|------|------|----|
| 名前・メール・電話 | 基本連絡先 | 田中太郎 / tanaka@gmail.com |
| **流入元（source）** | どこから来たか | ゴールドオンライン / Meta広告 |
| **商材（product）** | 何に興味があるか | B-1（民泊パッケージ） / A②（HAKU MATCH） |
| **ステータス** | どの段階か | 未対応 / アプローチ中 / 面談済 / 成約 / 失注 |
| **熱度（heat）** | 見込み度合い | A（高）/ B（中）/ C（低） |
| **担当者** | 誰が担当するか | 西出 / 松村 |
| 最終マーケアタック日 | 最後に連絡した日 | 2026-06-10 |
| 最終コンタクト日 | 最後に直接話した日 | 2026-06-12 |
| オプトアウト | 配信停止フラグ | true/false |

## 自動メッセージシーケンス（B-1 セミナー型）

申込から成約までの自動フォローアップ：

```
【申込直後】
  → メール: セミナー詳細 + LINE登録QR + 面談予約リンク（TimeRex）
  ※申込の熱が一番高い瞬間を逃さない

【セミナー前日 18時】
  → LINE: リマインド + セミナーURL

【セミナー当日 1時間前】
  → LINE: 直前リマインド

      ↓ セミナー当日 ↓

【出席者 → 終了後30分】
  → メール: セミナーまとめ + 個別面談予約リンク

【欠席者 → 2時間後】
  → メール: 録画URL + 面談予約リンク（「見逃した方へ」）

【面談未予約 → 3日後】
  → LINE: やわらかいフォロー

【面談未予約 → 7日後】
  → メール: 成功事例紹介 + 再案内
```

**重要**: LINEは月1,000通の上限があるため、開封率が高いリマインド（前日・当日）に絞って使用。それ以外はメールで対応。

## 追客アラート（Slackへ自動通知）

毎朝8時に全リードをチェックし、フォローが必要な人をSlackに通知します。

### デフォルトルール（随時カスタマイズ可能）

**ルール①: 熱度A/B × 7日間未アタック**
```
条件: 熱度がAまたはB かつ 最終マーケアタックから7日以上経過
     かつ ステータスが「成約」「失注」以外
通知内容: 
  🔥 【追客アラート】田中太郎さん（A熱）
  最終アタック: 8日前
  電話: 09012345678
  流入: ゴールドオンライン
```

**ルール②: 全リード × 14日間未アタック**
```
条件: ステータスが「成約」「失注」以外 かつ 14日以上アタックなし
通知内容:
  ⚠️ 【要フォロー】佐藤花子さん
  最終アタック: 15日前 / ステータス: アプローチ中 / 熱度: C
```

### ルールのカスタマイズ方法
Supabase の `alert_rules` テーブルに条件を追加・変更するだけ。  
例えば「セミナー参加者で面談未予約の人に3日後に通知」「特定の流入元だけ先に追客」なども設定可能。

## 既存リードの取り込み

現在Google Sheetsに散在している約15タブのリードデータを、**元データに一切手を加えず**、統一フォーマットでSupabaseに取り込みます。

### 取り込み対象のシート
- タスワンリード情報（統合）
- 泊マッチMeta
- その他13タブ

### 名寄せ（重複排除）のルール
1. メールアドレスが同じ → 同一人物として統合
2. 電話番号が同じ → 同一人物として統合
3. どちらもない → スキップ（ログに記録）

### 取り込み後にできること
- 流入元・熱度・ステータスでフィルタリング
- 条件を絞り込んでSMS・メールを一斉送信
- 最終アタック日でソートして追客優先順位をつける

## 一斉送信の使い方（Phase 2完成後）

### 操作の流れ
```
1. GAS WebApp（ブラウザで開く）にアクセス
2. 絞り込み条件を設定
   例: 「B-1 × 熱度A/B × 未対応 × 最終アタック30日以上前」
3. 対象リストのプレビューを確認（件数・名前一覧）
4. メッセージ内容を入力（テンプレートから選択可）
5. 「ドライラン」で送信内容を確認（実際には送らない）
6. 「送信実行」→ 全員に一斉送信
7. 送信ログが自動記録される
```

### 絞り込みの例
| 目的 | 条件 |
|------|------|
| 休眠リードへの再アプローチ | 最終アタック60日以上 × 失注以外 |
| 直近セミナー欠席者へのフォロー | 欠席フラグ × 直近7日以内 |
| 高意向未成約リードへのプッシュ | 熱度A × 面談済 × 未成約 |
| 流入元別の施策効果測定 | source = ゴールドオンライン × 申込30日以内 |

## 効果測定（Phase 2〜4で順次実装）

### 計測できるようになる指標

| 指標 | 計算式 | 目標 |
|------|--------|------|
| セミナー出席率 | 出席者 ÷ 申込者 | 50% → 70%に改善 |
| 面談予約率 | 面談予約者 ÷ 出席者 | 現状測定から開始 |
| 成約率 | 成約 ÷ 面談実施 | 測定から開始 |
| 流入元別コンバージョン | 媒体別の成約数 | ゴールドオンライン vs Meta比較 |
| メール開封率 | 開封数 ÷ 送信数 | 目標30%以上 |
| SMS返信率 | 返信数 ÷ 送信数 | 目標5%以上 |

### AIによる改善提案（Phase 4〜）
蓄積データをもとにClaude APIが毎週分析し：
- 「成約率の高い流入元TOP3」
- 「開封率が低い件名パターン」
- 「最適な追客タイミング」
- 「次の一手として推奨するメッセージ」

をSlackのレポートとして自動配信します。

## 営業担当が覚えること（Phase 1完成後）

### 毎朝やること
1. Slack の `#sales-alerts` を確認
2. アラートが来たリードに電話・メール

### ステータス更新のルール
面談後、Supabaseの該当リードを更新:
- 面談した → `面談済`
- 成約した → `成約`
- 見込みなし → `失注`
- 連絡取れた → `last_contacted_at` を今日の日付に更新

### 配信停止の処理
「もう連絡しないで」と言われたリード:
- `opted_out = true` に変更
- システムが自動で全チャネルからの配信を止める

## 今後の拡張予定（B-1以外への応用）

このシステムは**複数商材に対応できる設計**になっています：

| 商材 | 対応状況 |
|------|---------|
| **B-1** 民泊パッケージ | ✅ Phase 1でメイン実装 |
| **A②** HAKU MATCH | 🔲 Phase 2で追加（セミナー型→問い合わせ型に切替） |
| **B-2** 民泊コンサル | 🔲 B-1の仕組みをそのまま流用 |

リードDBの `product` カラムで商材を識別し、シーケンスも商材別に設定できます。
