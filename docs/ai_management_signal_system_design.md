## 1. システム概要

### 目的（短期）
Gmail・Slack・会議録（tl;dv・Plaud AI）・MiiTel・OTAレビューから**経営上の重要シグナル**（タスク・アイデア・リスク）を自動抽出し、**CEO Daily Briefing（毎朝3段階）**・**Slack Bot（社員Q&A）**・**KPIダッシュボード（Looker Studio）**で経営判断を支援する。

### 目的（最終）
経営判断の代替AI + 部門AI統合の基幹OS → **社員全員が創造的業務に集中する組織**を実現する。

### 対象ユーザー
- **社長:** CEO Daily Briefing（30秒/3分/深掘り版）、全レポート、KPIダッシュボード
- **管理職:** KPIダッシュボード、レポート閲覧
- **社員全員:** Slack Bot（施設情報・FAQ・マニュアル即時回答）
- **各担当者:** 自分に関連するタスクのみ通知

### 主要機能
1. **緊急アラート（毎時）:** キーワード検知時に即時DM通知
2. **CEO Daily Briefing（毎朝8時）:** 30秒版/3分版/深掘り版の3段階 ← 新規
3. **夕レポート（毎夕17時）:** 当日8:00〜16:59分析
4. **Slack Bot（24時間稼働）:** 社員Q&A + 自己改善ループ ← 新規
5. **Slackメンション監視:** 社長宛メンションを自動タスク化
6. **Looker Studio KPIダッシュボード:** 施設別レビュー推移・清掃不備・タスク完了率 ← 新規
7. **GitHub Vault:** MD形式の知識基盤（FAQ・マニュアル・Signal保存） ← 新規
8. **Webアプリ:** タスク管理・社員マスタ管理・設定・Plaud AI分析

## 4. 担当者管理

### 4.1 担当者判定フロー（段階的自動化）

**Phase 3: 半自動配分（人間確認あり）**
```

  ↓
AI担当者判定（社員マスタと照合）
  ├─ 個人名検知 → 候補として提案
  └─ 不明 → 部門ロールのデフォルト担当者を提案
  ↓
Webアプリで人間が確認・承認
  ├─ 追加・削除
  └─ 部門変更
  ↓
確定後
  ├─ Slack DM通知
  └─ Googleカレンダー登録
```

**Phase 7: 完全自動配分（最終目標）**
```
全データソース（Gmail・Slack・tl;dv・Plaud AI・MiiTel・OTA）
  ↓
Claude API 一括分析
  ├ タスク抽出
  ├ 担当者判定（過去履歴学習・業務量考慮）
  ├ 優先度判定（過去データ学習）
  ├ 期限判定（キーワード + 文脈理解）
  └ 信頼度スコア算出（0-100%）
  ↓
【信頼度80%以上】
  自動配分・自動通知・自動カレンダー登録
  （人間の承認不要）
  
【信頼度50-79%】
  社長に確認依頼（Slack #ai-確認）
  ✅承認 → 確定
  ❌却下 → 破棄 or 再割り当て
  
【信頼度50%未満】
  タスク候補として保存（週次レビュー）
```

**効果:**
- Phase 3: タスク配分作業 50%削減
- Phase 7: タスク配分作業 ほぼゼロ（95%自動化）

## 5. Webアプリ（GAS）

### 5.1 画面構成

#### タブ1: ダッシュボード
- 最新の朝レポート・夕レポート表示
- 未完了タスク数・今日の期限タスク数
- 最近のアイデア（上位5件）

#### タブ2: タスク管理
| 列 | 内容 | 操作 |
|---|------|------|
| 優先度 | 高/中/低 | 編集可能 |
| タスク内容 | AI抽出 | 編集可能 |
| 担当者 | AI提案 | 追加・削除可能 |
| 期限 | AI推定 | 編集可能 |
| ソース | Gmail/Slack/tl;dv/Plaud AI | 表示のみ |
| ソース詳細 | tl;dv: 2026-05-21 会議<br>Plaud AI: 非公開 | 表示のみ |
| ステータス | 未着手/進行中/完了 | 変更可能 |
| カレンダー | 未登録/登録済 | 承認ボタン |

**機能:**
- フィルタ（担当者・ステータス・期限・ソース）
- ソート（優先度・期限）
- カレンダー登録承認
  - 人間確認後に承認ボタンをクリック
  - 複数担当者：オーナー承認+ゲスト招待
  - 承認後：Googleカレンダー自動登録 + Slack DM通知

#### タブ3: アイデア管理
- リスト表示（日付・ソース・アイデア内容）
- ステータス変更（保留/検討中/実行/却下）

#### タブ4: Plaud AI分析（社長専用・Phase 4）
- テキストエリア：Plaud AI文字起こしを貼り付け
- 「分析」ボタン：Claude APIでリアルタイム分析
- 結果表示：抽出されたタスク・アイデア
- 「Sheetsに保存」ボタン
- **原文は保存されない**

#### タブ5: 設定
**4-1. 社員マスタ管理**
- 社員追加・編集・無効化
- 部門ロール設定

**4-2. アラートキーワード管理**
- キーワード追加・削除
- 有効/無効切替

**Slackメンション機能の詳細:**

#### データフロー
```
【朝8時 / 夕17時】

前日17:00〜当日7:59のデータ収集
├ Gmail履歴
├ tl;dv要約

  ↓

  ↓

```

#### AI判定項目
- **タスクタイトル:** メンション内容を要約（例: 「B社提案書作成」）
- **優先度:** 「至急」「緊急」→高、通常→中
- **期限:** 
  - 明示的: 「明日まで」→ 2026-05-22
  - 推測: 「至急」→ 翌営業日
  - 不明: 「要確認」フラグ + 背景色黄色

#### タスク重複防止
- GmailとSlackで同じ依頼が来た場合、Claude APIが検知
- 既存タスクの「補足情報」欄に追記
- 新規タスクとしては登録しない

#### タスク化判定基準（Claude APIプロンプト）
以下に該当する場合のみタスクとして抽出：
- 依頼・質問・確認・報告のいずれかを含む
- 具体的なアクションが必要
- 挨拶のみ・感謝のみのメンションは除外

**テスト期間:** 1週間（誤タスク化率を測定、目標: 10%以下）

**Phase 4移行判断:**
- 誤タスク化率が許容範囲内
- 部署宛メンション対応の必要性を評価

## 7B. Slack Bot 自己改善ループ（Phase 4-5実装）

### 基本フロー

```
社員がBotに質問（例: 「清掃チェックリストはどこ？」）
  ↓
Slack Bot受信（Zapier Trigger）
  ↓
権限判定（User IDで社員/管理職/社長を判別）
  ↓
GitHub Vault検索（Claude API）
  ├ 権限に応じたフォルダ内を検索
  └ FAQ・マニュアルから最適な回答を生成
  ↓
【答えが見つかった】
  Slackに回答を返す
    ↓
  社員が👎リアクション？
    ├ Yes: 誤回答として記録（Sheets保存）
    │       → 週次で修正候補生成（Claude API）
    │       → #ai-承認 チャンネルに投稿
    │       → 社長が✅承認 → GitHub Vault更新
    └ No: 正常終了
  
【答えが見つからない】
  「現在回答できません。記録しました」と返す
    ↓
  質問を自動記録（Sheets「未回答質問リスト」）
    ↓
  週次でFAQ候補生成（Claude API）
    ├ 回答ドラフト作成
    └ 追加すべきFAQカテゴリ提案
    ↓
  #ai-承認 チャンネルに投稿
    ↓
  社長が✅承認 → GitHub Vault更新
```

### 自己改善サイクル（週次）

**毎週月曜 9時（自動実行）:**

1. **未回答質問の集計**
   - Sheets「未回答質問リスト」から過去1週間分を取得
   - 重複質問を統合

2. **FAQ候補生成（Claude API）**
   - 質問内容を分析
   - 回答ドラフト作成
   - 追加すべきフォルダ（public/internal/management）提案

3. **修正候補生成（Claude API）**
   - 👎リアクションがついた回答を分析
   - 正しい回答案を生成
   - 修正理由を説明

4. **承認依頼投稿（Slack #ai-承認）**
   ```
   【週次FAQ改善提案】2026-05-27
   
   ━━━━━━━━━━━━━━━━━━━━
   📝 新規FAQ候補（3件）
   ━━━━━━━━━━━━━━━━━━━━
   
   Q1: 「緊急時の連絡先は？」（3回質問あり）
   A案: [Claude生成回答]
   追加先: faq/internal/emergency_contacts.md
   ✅ 承認 / ❌ 却下
   
   Q2: [...]
   
   ━━━━━━━━━━━━━━━━━━━━
   🔧 回答修正候補（2件）
   ━━━━━━━━━━━━━━━━━━━━
   
   Q: 「清掃チェックリストはどこ？」
   現在の回答: [誤回答]
   修正案: [正しい回答]
   理由: 社員が👎リアクション
   ✅ 承認 / ❌ 却下
   ```

5. **承認後の自動反映**
   - 社長が✅リアクション
   - Zapier検知 → GitHub Vault更新（GitHub API）
   - Sheets「未回答質問リスト」を削除

## 10. リスクと対策

### リスク1: tl;dvメール取得失敗
**対策:** n8n監視エラー時にSlack通知 → 管理者が手動投稿

### リスク2: Plaud AI投稿忘れ
**対策:** 朝レポート生成時にデータ欠損を検知 → Claude APIが「データなし」と明記

### リスク3: Claude APIタイムアウト
**対策:** データ量が多い場合は分割処理（朝レポート・夕レポート分離済み）

### リスク4: 担当者誤判定
**対策:** AI提案を必ず人間が確認・承認するフロー

### リスク5: Slackメンションの誤タスク化（Phase 3以降）
**原因:** 雑談中のメンション（「@西出社長、お疲れ様です」）もタスク化される
**対策:**
- Claude APIプロンプトに「タスク判定基準」を明記
- 挨拶・感謝のみのメンションは除外
- 誤タスク化率を測定（目標: 10%以下）

### リスク6: タスク重複登録（Phase 3以降）
**原因:** GmailとSlackで同じ依頼が複数回送られる、またはtl;dvとPlaud AIで同じ会議が記録される
**対策:**
- Claude APIでタスク重複検知（キーワード70%一致・担当者同一・期限3日以内重複）
- 最も詳細な情報源を優先（通常はPlaud AI）
- ソース列に「tl;dv + Plaud AI」と記載
- 既存タスクに補足情報として追記
- Webアプリで人間が最終確認

### リスク7: Plaud AIプライバシー漏洩（Phase 3以降）
**原因:** Plaud AI原文が誤って公開される
**対策:**
- 原文はどこにも保存しない（専用Webインターフェース方式）
- タスク・アイデアのみSheets保存
- ソース詳細列に「Plaud AI: 非公開」と表示
- Webアプリでのアクセス制御（社長のみPlaud AIタブにアクセス可能）
